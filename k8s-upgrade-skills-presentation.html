Phase 0 — 가장 중요한 단계

결정론적 사전 검증

$ python3 scripts/gate_check.py \
--cluster-name my-cluster \
--current-version "1.34" \
--target-version "1.35"
exit 0 — Gate OPEN
exit 1 — BLOCKED (즉시 중단)
exit 2 — WARN (사용자 확인)

스크립트가 Gate를 판단 — LLM이 bypass 불가.
audit.log에 결과 자동 기록, LLM은 읽기만 가능.

08 / 16
Why Not LLM Only?

왜 LLM에 다 맡기면 안 되는가

LLM 단독

  • • Hallucination으로 검증 결과 조작 가능
  • • 프롬프트 변경 시 검증 로직 변동
  • • "PASS로 판단했습니다" — 근거 불투명
  • • 감사 추적 불가능

gate_check.py

  • • Exit code 기반 — 결과 조작 불가
  • • 동일 입력 → 동일 결과 보장
  • • kubectl/aws CLI 실제 데이터 기반
  • • audit.log 자동 기록 — 추적성 확보

Phase 0 = 스크립트가 판단 (신뢰도 100%)  |  Phase 1~7 = LLM 해석 (보조적 판단, 수동 교차 검증 권장)

17개 검증 규칙

Common 4개
  • ● COM-001 클러스터 상태 + 노드 Ready
  • ● COM-002 버전 호환성 (minor +1)
  • ● COM-002a kubelet 버전 skew
  • ● COM-003 Add-on 호환성
Workload Safety 6개
  • ● WLS-001 PDB 차단 (disruptionsAllowed=0)
  • ● WLS-002 단일 레플리카 위험
  • ● WLS-003 PV AZ 고정
  • ● WLS-004~006 로컬스토리지 · Job · 토폴로지
Capacity 3개
  • ● CAP-001 노드 CPU/MEM 여유분
  • ● CAP-002 OOMKilled · CrashLoop 감지
  • ● CAP-003 서브넷 가용 IP (Surge)
Infrastructure 4개
  • ● INF-001 Terraform state drift
  • ● INF-002 AMI 가용성 (SSM 조회)
  • ● INF-003 Karpenter CRD 호환성
  • ● INF-004 Terraform recreate 감지
● CRITICAL — 즉시 중단 ● HIGH — 사용자 확인 ● MEDIUM/LOW — 보고만

Roadmap — Priority 1

지능형 관측성 및 Gate 강화

Service-Aware Gate

현재: 노드 Ready 상태만 확인
목표: EndpointSlice 가용 IP + 실제 헬스체크 응답 기반 Gate 판단.
노드가 Ready여도 Ingress/Service 엔드포인트 전파 지연으로 5xx 발생 가능 → 이를 방지

병렬 Sub-Agent Drain 모니터링

IaC 실행과 동시에 Sub-Agent가 kubectl get events -w 실시간 감시.
PDB 교착, Graceful Termination 실패 등 드레인 지연 원인 즉각 감지 + 인터랙티브 대응 가이드

자동 RCA 리포트

Gate 미통과 시 클러스터 상태 스냅샷 자동 저장 (Events, Pod Status, IaC Plan).
AI Agent가 즉각적인 근원 분석(Root Cause Analysis) 리포트 생성 — 수동 로그 수집 불필요
