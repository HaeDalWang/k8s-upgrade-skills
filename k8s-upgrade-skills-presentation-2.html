06 / 14
Why Not LLM Only?

왜 LLM에 다 맡기면 안 되는가

LLM 단독

  • • Hallucination으로 검증 결과 조작 가능
  • • 프롬프트 변경 시 검증 로직 변동
  • • "PASS로 판단했습니다" — 근거 불투명
  • • 감사 추적 불가능

gate_check.py + phase_gate.py

  • • Exit code 기반 — 결과 조작 불가
  • • 동일 입력 → 동일 결과 보장
  • • kubectl/aws CLI 실제 데이터 기반
  • • Phase 0~7 전 구간 결정론적 Gate

Phase 0 = gate_check.py  |  Phase 2~7 = phase_gate.py  |  Phase 1 = LLM (IaC 변경)

07 / 14
LLM의 역할

데이터는 있는데 "이게 문제인지"가
상황마다 다를 때

Terraform plan에 replace 감지

DB replace면 CRITICAL, SG replace면 괜찮을 수도
→ 고객사 정책에 따라 판단이 달라짐

노드 사용률 85% — 어떤 노드냐

On-demand 노드 85% → drain 시 여유 없음, 위험
Karpenter 노드 85% → drain 시 자동 추가, 정상
스크립트는 % 수치만 봄. 노드 관리 주체는 레이블 보고 판단

Job이 6시간째 실행 중

배치 잡이면 정상, 일반 잡이면 이상
→ 잡 이름·레이블을 보고 판단해야 함

임계값을 코드에 박으면 False Positive 폭탄.
스크립트가 데이터를 모으고, LLM이 맥락을 읽는 구조.

11 / 14
Roadmap — Priority 1

지능형 관측성 및 Gate 강화

Service-Aware Gate

현재: 노드 Ready 상태만 확인
목표: EndpointSlice 가용 IP + 실제 헬스체크 응답 기반 Gate 판단.
노드가 Ready여도 Ingress/Service 엔드포인트 전파 지연으로 5xx 발생 가능 → 이를 방지

병렬 Sub-Agent Drain 모니터링

IaC 실행과 동시에 Sub-Agent가 kubectl get events -w 실시간 감시.
PDB 교착, Graceful Termination 실패 등 드레인 지연 원인 즉각 감지 + 인터랙티브 대응 가이드

자동 RCA 리포트

Gate 미통과 시 클러스터 상태 스냅샷 자동 저장 (Events, Pod Status, IaC Plan).
AI Agent가 즉각적인 근원 분석(Root Cause Analysis) 리포트 생성 — 수동 로그 수집 불필요
